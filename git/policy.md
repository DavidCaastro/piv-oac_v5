## Commit Message Policy

All commits must follow Conventional Commits format.
Language: English only (no Spanish commit messages).

```
<type>(<scope>): <short description>

[optional body — 72 char wrap]

[optional footer: Breaking-Change, Refs, Co-authored-by]
```

### Allowed Types

| Type | When to use |
|---|---|
| `feat` | New capability or behavior |
| `fix` | Bug correction |
| `refactor` | Code restructure with no behavior change |
| `test` | Add or modify tests only |
| `docs` | Documentation only |
| `chore` | Build, tooling, dependency updates |
| `ci` | CI/CD workflow changes |
| `perf` | Performance improvement |

### Scope Examples

| Scope | Meaning |
|---|---|
| `auth` | Authentication domain |
| `sdk` | Framework SDK changes |
| `gate` | Gate logic modifications |
| `engram` | Memory layer changes |
| `spec` | Spec file modifications |

### Commit Frequency (Specialist Agents)

- One commit per logical unit — not one commit at the end of the task
- Each commit must leave the branch in a buildable, test-passing state
- Stubs raised as `NotImplementedError` are committed before implementation begins
- Pre-commit hooks must pass before each commit (no `--no-verify`)

## Branch Naming Policy

Branch types (Directive / Artifact) are defined in `git/topology.md §Branch Types`.
This table governs naming patterns, ownership, and lifecycle only.

### Directive Branches

| Branch | Owner | Lifecycle |
|---|---|---|
| `architect` | Human (framework author) | Permanent — framework development |
| `sec_ops` | Human (security lead) | Permanent — security contracts and gate policies |
| `piv-directive` | SDK only | Permanent (orphan) — AuditAgent engram atoms |

### Artifact Branches

| Pattern | Owner | Lifecycle |
|---|---|---|
| `main` | Human team | Permanent |
| `staging` | SDK / CI | Permanent |
| `feature/<task-id>/` | DomainOrchestrator | PHASE 5 → Gate 2b |
| `feature/<task-id>/expert-<N>` | Specialist Agent N | PHASE 5 → Gate 1 |
| `fix/<issue-id>/` | MasterOrchestrator | Hotfix cycle |

**Task ID format:** `<domain>-<NNN>` — e.g. `auth-001`, `payments-042`
**Expert number:** sequential integer starting at 1 per task

## What Agents May and May Not Do

### Specialist Agents (L2)

| Allowed | Prohibited |
|---|---|
| Commit to own subbranch `feature/<task-id>/expert-N` | Push to any other branch |
| `git add`, `git commit` within own worktree | `git merge`, `git rebase` |
| `git status`, `git diff` in own worktree | Read another expert's worktree |
| Use pre-commit hooks | Bypass hooks with `--no-verify` |
| — | Push or commit to any directive branch (`architect`, `sec_ops`, `piv-directive`) |

### Domain Orchestrators (L1.5)

| Allowed | Prohibited |
|---|---|
| Create `feature/<task-id>/` and subbranches | Push to `staging` or `main` |
| Submit PR from task branch to staging | Force push any branch |
| Prune merged worktrees | Delete `piv-directive` |
| — | Push, commit, or open PRs against any directive branch |

### AuditAgent (L1)

| Allowed | Prohibited |
|---|---|
| Write to `piv-directive` (engram atoms) | Push to `main` or `staging` |
| Append `.piv/` checkpoint files | Overwrite existing engram atoms |
| — | Write to `architect` or `sec_ops` |

### Cross-type rule (all agents)

**No agent may ever commit to, push to, or open a PR against a directive branch
from within an artifact branch context.**
Changes to directive branches must be routed through `main` and validated by
the framework pipeline before a human merges them into the directive branch.
See `git/topology.md §Working on a directive branch`.

## Pull Request Policy

| Target branch | Required checks | Human approval |
|---|---|---|
| `feature/<task-id>/` (from expert subbranch) | `piv/pre-merge` (Gate 1) | Not required |
| `staging` (from task branch) | `piv/gate2b` (Gate 2b) | Not required |
| `main` (from staging) | `piv/staging-gate` (Gate 3) | **Required** (1 reviewer) |

**PR title format:** `<type>(<scope>): <short description>` — same as commit type.
**PR body:** auto-generated by AuditAgent from TechSpecSheet + gate verdicts.

## Merge Strategy

| Scenario | Strategy | Rationale |
|---|---|---|
| Expert subbranch → task branch | Squash merge | One clean commit per expert in task history |
| Task branch → staging | Merge commit | Preserves task boundary in staging history |
| Staging → main | Merge commit | Full audit trail required |
| Any hotfix | Merge commit | Full traceability |

No `git rebase` on shared branches.
No `git push --force` anywhere.
